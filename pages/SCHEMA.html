Wiki Schema

Domain

AI research and adjacent AI engineering knowledge: models, labs, techniques, benchmarks, tooling, evaluation, inference, training, productization, and important people/organizations. The wiki is intended to compound over time into a practical research and operations knowledge base, not a raw clipping archive.

Conventions

  • File names: lowercase, hyphens, no spaces (e.g., transformer-architecture.md)
  • Every wiki page starts with YAML frontmatter
  • Use wikilinks to link between pages (minimum 2 outbound links per page)
  • When updating a page, always bump the updated date
  • Every new page must be added to index.md under the correct section
  • Every action must be appended to log.md
  • Raw source files live under raw/ and are immutable after capture
  • Prefer concise, scannable pages; move long derivations or deep dives into separate pages
  • Use direct source-based claims; if a claim is uncertain, label it explicitly

Page Thresholds

  • Create a page when an entity/concept appears in 2+ sources OR is central to one source
  • Add to an existing page when a source mentions something already covered
  • Don't create a page for passing mentions, minor details, or things outside AI research scope
  • Split a page when it exceeds ~200 lines; break into sub-topics with cross-links
  • Archive a page when its content is fully superseded; move to _archive/, remove from index

Entity Pages

One page per notable entity. Include:

  • Overview / what it is
  • Key facts and dates
  • Relationships to other entities (wikilinks)
  • Source references

Entity categories commonly include people, companies, labs, open-source projects, products, models, and benchmarks when a dedicated entity page is more natural than a concept page.

Concept Pages

One page per concept or topic. Include:

  • Definition / explanation
  • Current state of knowledge
  • Open questions or debates
  • Related concepts (wikilinks)

Common concept topics include training methods, evaluation patterns, inference systems, scaling laws, agent workflows, safety methods, and benchmark design.

Comparison Pages

Side-by-side analyses. Include:

  • What is being compared and why
  • Dimensions of comparison (table format preferred)
  • Verdict or synthesis
  • Sources

Examples: model-vs-model, benchmark-vs-benchmark, lab strategy comparisons, inference stack trade-offs.

Query Pages

Use queries/ for substantial answers worth preserving, such as:

  • deep research syntheses
  • recurring user questions
  • multi-source comparisons that would be expensive to reconstruct

Do not file trivial lookups.

Update Policy

When new information conflicts with existing content:

  1. Check dates — newer sources generally supersede older ones
  2. If genuinely contradictory, note both positions with dates and sources
  3. Mark the contradiction in frontmatter: contradictions: [page-name]
  4. Flag for user review in the lint report

Default Structure Hints

  • entities/ for labs, companies, products, notable researchers, named models, datasets, benchmarks
  • concepts/ for methods, abstractions, research themes, technical mechanisms
  • comparisons/ for structured side-by-side analyses
  • queries/ for reusable syntheses and non-trivial question answers

Preferred Style

  • Lead with a compact summary section
  • Prefer bullets and short sections over long prose
  • Keep claims attributable to sources
  • Include open questions where the field is unsettled
  • Link outward aggressively so the wiki becomes navigable as it grows
